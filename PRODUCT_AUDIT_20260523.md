# Product Audit Report — КорнДок — 2026-05-23

## Platform purpose

КорнДок — корпоративная система хранения и семантического поиска по документам.
Решает проблему «где найти нужный документ» в условиях, когда файлы разбросаны по почте,
дискам и мессенджерам. Основной пользователь — рядовой сотрудник компании, которому нужно
быстро найти регламент, договор или инструкцию без знания точного названия файла.
Ценностное предложение: загрузил документ → через 30 сек он нашёлся по фразе из середины текста.

## User roles overview

| Роль | Кто это? | Что может? | Ключевые ограничения |
|------|----------|-----------|----------------------|
| `user` | Обычный сотрудник | Загружать, искать, скачивать, шарить свои документы | Видит только доступные ему документы |
| `admin` | Системный администратор | Всё, включая просмотр ВСЕХ документов и управление пользователями | Не может деактивировать свой аккаунт, понизить свою роль |

> **Важно**: нет промежуточных ролей (manager, department head). Только user и admin.
> Нет самостоятельной регистрации — все аккаунты создаёт только admin.

## Document permission levels — что каждый уровень реально даёт

| Level | Скачать файл | Превью текста | Виден в поиске | Управлять правами | Удалить | Редактировать метаданные |
|-------|-------------|--------------|----------------|-------------------|---------|--------------------------|
| viewer | ✅ API+UI | ✅ API+UI | ✅ | ❌ API блокирует | ❌ API блокирует | ❌ API блокирует |
| editor | ✅ API+UI | ✅ API+UI | ✅ | ❌ API блокирует | ❌ API блокирует | ✅ API+UI |
| owner | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| admin | ✅ (все документы) | ✅ (все документы) | ✅ (все документы) | ✅ | ✅ | ✅ |

**Уточнение по owner в UI**: кнопки «Поделиться» и «Удалить» в DocumentsPage показываются
только если `doc.uploaded_by === currentUserId` или `isAdmin`. Если пользователю была
**выдана** роль owner другим пользователем (через DocumentPermission), UI не покажет эти кнопки,
хотя API будет работать корректно. Это расхождение UI-API описано ниже.

## Page-by-page analysis

### / — Главная страница (HomePage)

**Что показывается:**
- Приветствие с именем пользователя (из профиля)
- Для admin: 4 системные статистики (всего документов, проиндексировано, поисков сегодня, пользователей)
- Быстрые действия: Найти / Загрузить / ИИ-ассистент
- Последние 5 документов из `GET /api/documents/?page=1&limit=5`

**Откуда данные:** Profile API + Admin stats API (только для admin) + Documents API (первые 5)

**Проблемы:**
- Последние документы — это НЕДАВНО ЗАГРУЖЕННЫЕ В СИСТЕМУ, а не «недавно открытые пользователем».
  Если в системе много чужих документов, пользователь видит активность коллег, а не свою.
  Нет связи между «последние документы» и тем, что пользователь реально просматривал.
- Для нового пользователя без документов: показывается пустое состояние «Документов пока нет».
  Это нормально, но нет подсказки «попросите коллегу поделиться документом с вами».

### /search — Страница поиска (SearchPage)

**Что показывается:** поисковая строка + фильтры (тип файла, даты), карточки результатов

**Откуда данные:** `POST /api/search/` с фильтром по правам доступа в Qdrant

**Чьи документы ищутся:** корректно — только документы, к которым у пользователя есть хотя бы viewer-доступ.
Фильтр строится в `_build_qdrant_filter()` и учитывает: загруженные пользователем, org-уровневые разрешения,
личные разрешения.

**Найденные проблемы:**

1. **Пустое состояние при первом входе:** если у пользователя нет ни одного доступного документа,
   поиск вернёт пустой список. Нет подсказки «попросите коллегу поделиться документом».

2. **Фильтры (поле `folder_path`):** фильтр папки — точное совпадение. Если документы в `/юридический/договоры/`,
   поиск по `/юридический/` не найдёт их. Не ожидают этого поведения пользователи.

3. **Нет счётчика «всего найдено» до дедупликации:** поле `total` в ответе возвращает количество
   результатов ПОСЛЕ дедупликации и фильтров, что правильно, но не показывает сколько чанков
   совпало изначально.

4. **Нет кнопки «Поделиться» в карточке результата:** у владельца документа нет быстрого
   способа поделиться прямо из поиска. Нужно идти в Documents page.

5. **Similarity score без объяснений:** пользователь видит процент релевантности (82%), но не понимает,
   что это значит. Нет тултипа.

### /documents — Страница документов (DocumentsPage)

**Что показывается:** таблица/сетка документов с фильтрами, сортировкой, переключением режима вида

**Откуда данные:** `GET /api/documents/` — фильтрует по правам доступа

**Чьи документы появляются здесь:**
- Документы, загруженные самим пользователем (`uploaded_by == user.id`)
- Документы организации, в которой состоит пользователь (`doc.org_id in user_org_ids`)
- Документы, для которых у пользователя есть явная запись в `document_permissions`

**⚠️ Критическая проблема №1 — Ghost List Bug:**

Документы с `org_id` принадлежащим организации пользователя появляются в списке,
но **`check_document_access` не даёт автоматически доступ** только на основании `org_id` документа.

Сценарий:
1. Алиса загружает документ, указывает `org_id=5` (Юридический отдел)
2. Алиса получает личную запись `DocumentPermission(user_id=alice, level=owner)` — НЕ организационную
3. Боб состоит в org_id=5
4. Боб **видит документ** в своём списке (список фильтрует по `doc.org_id in user_org_ids`)
5. Боб **получает 403** при попытке открыть, скачать или найти документ в поиске

Документ-призрак: виден в списке, но недоступен. Для пользователя это баг — «почему я вижу
документ, но не могу его открыть?»

**Источник расхождения:** `list_documents` использует `Document.org_id.in_(user_org_ids)` как критерий
видимости, а `check_document_access` требует явной записи в `document_permissions`. Они не синхронизированы.

**⚠️ Критическая проблема №2 — UI не знает уровень доступа пользователя к каждому документу:**

`GET /api/documents/` возвращает `DocumentOut` без поля `access_level`. UI определяет,
кому показывать кнопки «Поделиться» и «Удалить», только через:
```
isAdmin || doc.uploaded_by === currentUserId
```

Следствие: пользователь, которому выдали уровень `owner` через `DocumentPermission`,
**не увидит** кнопки «Поделиться» и «Удалить» в UI, хотя API позволяет эти действия.

**Остальное:**
- Фильтры тип+статус работают через API (server-side), фильтр по названию и владельцу — client-side
  на уже загруженной странице. Это значит, что клиентский поиск по названию ищет только среди
  первых 20 документов, а не по всем.
- Переключение вид (list/grid) сохраняется в localStorage — хорошо.
- Пустое состояние: «Нет документов». Нет контекста — не понятно, нет вообще или нет с этим фильтром.

### /upload — Страница загрузки (UploadPage)

**Что показывается:** drag-and-drop зона, очередь файлов, поля Папка и Теги

**⚠️ Проблема №3 — нет выбора организации при загрузке:**

API `POST /api/documents/upload` принимает `org_id`, но UploadPage не имеет этого поля.
Пользователь не может загрузить документ сразу «для организации» — все загрузки идут как личные.
Чтобы документ стал виден через org, нужно потом вручную создать `DocumentPermission(org_id=...)` через Share.

**Остальное:**
- Теги и папка применяются ко ВСЕМ файлам в очереди одинаково. Нет per-file настроек.
- После загрузки файл уходит в "Ожидание" — BackgroundTaskMonitor отслеживает статус и
  уведомляет когда проиндексировано. Это хорошо работает.
- Нет ограничения на тип файла на уровне браузера — `accept=".pdf,.docx,.xlsx,.txt"` есть,
  но если человек перетаскивает файл не того типа, ошибка приходит только после загрузки.
  Можно проверять тип файла до загрузки.

### /ai — ИИ-ассистент (AIAssistantPage)

**Что показывается:** полноэкранный чат-интерфейс на базе @gravity-ui/aikit, история сессий слева

**Как работает:**
1. Пользователь пишет вопрос
2. Сервер определяет нужны ли инструменты (documents / faq / both / direct)
3. Если нужно — вызывает `search_documents` и/или `search_faq` через внутренний API
4. Инструменты используют auth_header пользователя → поиск только по его документам ✅
5. Ответ стримится по SSE

**Доступ к данным:**
- `search_documents` вызывает `POST /api/search/` с токеном пользователя → корректная фильтрация ✅
- `search_faq` вызывает `GET /api/search/faq` → публичные FAQ ✅
- Галлюцинации ограничены: ИИ получает текст найденных чанков и строит ответ на основе них.
  Но если документы не найдены — может ответить из общих знаний (нет жёсткого запрета)

**Проблемы:**
- Нет явного указания источников в ответе с кликабельными ссылками на документы.
  ИИ может написать «В документе Х сказано Y», но пользователь не может быстро перейти к документу.
- Переключатели "Документы" и "FAQ" правильно передаются как `disabled_tools`, но API
  пока не использует это поле (в коде nodes.py нет фильтрации по disabled_tools) — они отправляются
  на сервер, но эффекта нет.
- Нет ответа на вопрос «каков размер контекстного окна». Истории берётся только 20 последних
  сообщений (`_HISTORY_LIMIT = 20`), старые теряются.

### /faq — FAQ страница (FAQPage)

**⚠️ Критическая проблема №4 — двойная система FAQ:**

**FAQPage.tsx** читает вопросы и ответы из i18n-переводов (`t('q1')`, `t('a1')`, ..., `t('q14')`).
Это **статические жёстко прошитые** данные в коде фронтенда.

**Параллельно существует** полноценная динамическая система:
- `FAQItem` модель в PostgreSQL
- Admin-панель, вкладка FAQ: создание/редактирование/удаление FAQ-элементов
- Индексирование FAQ в Qdrant (отдельная коллекция `qdrant_faq_collection`)
- `GET /api/search/faq` — семантический поиск по FAQ
- ИИ-ассистент использует это через инструмент `search_faq`

Эти две системы **никак не связаны**. Если admin создаст новый FAQ через панель,
на странице `/faq` он не появится. Изменение содержимого FAQ требует правки кода фронтенда.

Публичный эндпоинт `GET /api/faq` (без авторизации) существует, но FAQPage его не вызывает.

## Access control — подтверждённые проблемы

- 🔴 **[documents.py:list_documents — Ghost List Bug]** — документы с `org_id` пользователя
  появляются в списке без права доступа. Severity: **functional** (пользователь видит документ
  которого не может открыть — создаёт путаницу и 403 ошибки).

- 🔴 **[DocumentsPage — access_level не передаётся из API]** — UI показывает кнопки Share/Delete
  только загрузившему и admin. Пользователи с выданным owner-правом не получают эти кнопки в UI.
  Severity: **functional** (функция работает через API, но недоступна через UI).

- 🟡 **[DocumentsPage — client-side поиск по первой странице]** — строка поиска по названию
  в DocumentsPage ищет по уже загруженным 20 документам, а не по всей базе.
  Severity: **functional** (пользователь с 50+ документами не найдёт документ с 21-й позиции).

- 🟡 **[organizations.py:lookup_organizations — открыт всем аутентифицированным]** —
  любой пользователь может искать ВСЕ организации в системе через `GET /api/organizations/lookup`.
  Это нужно для Share-диалога, но раскрывает структуру компании. Severity: **cosmetic** (имена org
  не конфиденциальны в корпоративном контексте, но стоит задуматься).

- 🟡 **[users.py:lookup_users — открыт всем аутентифицированным]** —
  любой пользователь может искать ВСЕХ активных пользователей по email/username.
  Severity: **cosmetic** (нужно для share-диалога).

- 🟡 **[admin.py — логин не логируется]** — `POST /api/auth/login` не создаёт AuditLog запись.
  Severity: **cosmetic** (для audit trail это пробел).

- 🟡 **[FAQPage.tsx — статические данные вместо API]** — FAQPage не вызывает `/api/faq`.
  Severity: **functional** (admin не может обновить FAQ через панель — изменения не видны пользователям).

- 🟡 **[AIAssistantPage — `disabled_tools` не применяется на сервере]** — переключатели
  инструментов передаются, но в коде агента не обрабатываются.
  Severity: **cosmetic** (UX feature не работает, но никакого вреда нет).

## User journey assessment

### Journey: Новый сотрудник, первый день

1. ❌ **Регистрация** — нет самостоятельной регистрации. Admin должен создать аккаунт вручную.
   Сотрудник не может сам завести учётку. Нет invite-flow.
2. ✅ **Вход** — логин по email+password работает. JWT сохраняется в localStorage.
3. ✅ **Главная страница** — приветствие, быстрые действия. Понятно.
4. ⚠️ **Поиск без документов** — пустые результаты. Нет подсказки «попросите коллегу поделиться».
5. ✅ **Загрузка документа** — работает, документ появляется в списке.
6. 🔴 **Документ сотрудника из его org виден в списке, но недоступен** — ghost list bug.

Вердикт: онбординг требует обязательного участия admin. Начать пользоваться самостоятельно невозможно.

---

### Journey: Сотрудник загружает документ и шарит его

1. ✅ **Загрузка PDF** — работает. Файл идёт в MinIO, задача в Celery.
2. ✅ **Статус "Индексирован"** — BackgroundTaskMonitor опрашивает каждые 3 сек, всплывает уведомление.
3. ❌ **Нет выбора организации при загрузке** — документ всегда личный. Org-привязка только через Share после загрузки.
4. ✅ **Поделиться через Share modal** — работает. Можно выдать viewer/editor/owner отдельному пользователю или организации.
5. ✅ **Коллега с viewer находит документ в поиске** — ✅
6. ✅ **Viewer может скачать и превью** — ✅
7. ✅ **Viewer не может редактировать метаданные** — API блокирует ✅

Вердикт: happy path работает хорошо. Главный gap — нельзя загрузить документ сразу «в организацию».

---

### Journey: Поиск и использование ИИ-ассистента

1. ✅ **Поиск** — семантический поиск с гибридным ranking (dense+sparse RRF) работает хорошо.
   Подсветка найденных фрагментов (`<mark>`) есть.
2. ✅ **Превью** — PDF открывается в iframe через presigned URL. Текстовые форматы показывают первые 3000 символов.
3. ✅ **ИИ отвечает только из доступных документов** — поиск через auth header пользователя.
4. ⚠️ **Нет кликабельных ссылок на источники** — ИИ говорит «в документе X», но нет перехода к документу.
5. ⚠️ **Переключатели инструментов не работают** — disabled_tools игнорируется на сервере.

Вердикт: поиск и ИИ работают хорошо. Пост-MVP улучшение — ссылки на источники в ответах ИИ.

## Biggest product logic questions

### 1. Admin document visibility

**Вопрос:** Admin видит все документы в системе, включая содержимое. Верно ли это?

**Аргументы «за»:** Admin должен управлять системой — переиндексировать, удалять, исправлять ошибки.

**Аргументы «против»:** Если бухгалтерия шарит документ только с CFO, а admin-аккаунт может его прочитать — это нарушение need-to-know. Особенно критично для НДА, персональных данных, зарплатных ведомостей.

**Текущая реализация:** `GET /api/admin/documents` возвращает все документы с метаданными и позволяет редактировать. `check_document_access` возвращает документ для admin без проверки. Admin может запустить переиндексацию и тем самым вызвать загрузку файла из MinIO.

**Рекомендация:** Ввести два вида admin — `system_admin` (управление платформой, без доступа к содержимому) и `content_admin` (полный доступ к документам). На MVP — хотя бы задокументировать это ограничение и убедиться, что admin-аккаунт — не личный аккаунт конкретного сотрудника.

---

### 2. Ghost List Bug — нужно принять решение об исправлении

**Вопрос:** Должен ли факт принадлежности документа к организации (org_id) автоматически давать
всем членам этой организации хотя бы viewer-доступ?

**Текущая реализация:** `org_id` на документе — это **аффилиация/тег**, а не право доступа.
Право доступа — отдельная запись в `document_permissions`. Это архитектурно чисто, но неинтуитивно.

**Рекомендация (вариант A):** При загрузке с `org_id` автоматически создавать `DocumentPermission(org_id=..., level="viewer")`. Тогда org_id означает «виден всей организации».

**Рекомендация (вариант B):** Убрать `org_id` из условий видимости в `list_documents`. Документ появляется в списке только если есть явный permission. org_id остаётся как тег/фильтр.

**Текущий список**: показывает документы обоих типов — это баг.

---

### 3. Нет самостоятельной регистрации

**Вопрос:** Должны ли новые сотрудники регистрироваться сами, или только через admin?

**Текущая реализация:** Только через admin. Нет registration-endpoint в API.

**Рекомендация:** Для корпоративной системы это нормально (controlled onboarding). Но нужна
хотя бы страница с объяснением «Обратитесь к администратору для получения доступа» вместо пустой формы логина.

---

### 4. FAQ — две несвязанные системы

**Вопрос:** FAQ — статические переводы или динамические записи в БД?

**Текущая реализация:** и то, и другое. Frontend FAQ — статика. Backend FAQ — динамика для ИИ.

**Рекомендация:** FAQPage должна вызывать `GET /api/faq` (публичный endpoint уже существует).
Тогда admin создаёт FAQ в панели → он появляется и на /faq странице, и в ИИ-поиске.

---

### 5. Передача владения документом

**Вопрос:** Что происходит с документом, если загрузивший его уволился?

**Текущая реализация:** `uploaded_by` остаётся неизменным (FK с ondelete=SET NULL, но сам факт загрузчика остаётся). Если аккаунт деактивирован — документ есть, но нет «живого» владельца. Другие люди могут иметь owner-право через DocumentPermission, но без понимания контекста.

**Рекомендация:** Нужна функция «передать владение» или назначения нового primary owner при деактивации аккаунта.

## Summary

### Работает как ожидается ✅
- Семантический поиск (hybrid dense+sparse, RRF fusion) с подсветкой результатов
- Загрузка, парсинг и индексирование документов (PDF, DOCX, XLSX, TXT)
- Системы прав доступа: viewer/editor/owner правильно применяются на уровне API
- ИИ-ассистент ищет только в документах, доступных пользователю
- Система организаций: создание, добавление участников
- Share-диалог: выдача/отзыв прав конкретным пользователям и организациям
- Превью документов (PDF через presigned URL iframe, текст — первые 3000 символов)
- Audit log для основных действий (кроме login)
- Тёмная/светлая тема, мультиязычность (RU/EN), уведомления

### Требует внимания ⚠️
- UploadPage не имеет поля для выбора org_id
- DocumentsPage не возвращает `access_level` из API → Share/Delete кнопки скрыты для granted owners
- Client-side поиск по названию в DocumentsPage работает только на текущей странице (20 записей)
- Переключатели инструментов ИИ (`disabled_tools`) не обрабатываются на сервере
- Нет кликабельных ссылок на источники в ответах ИИ
- Нет логирования событий login в audit_logs
- Нет раздела «Мои документы» с пагинацией на странице профиля
- Отсутствует autocomplete по папкам при загрузке

### Сломано или отсутствует с точки зрения продукта ❌
- **Ghost List Bug**: документы с `org_id` пользователя появляются в списке, но возвращают 403 при попытке доступа
- **FAQPage статическая**: пользователи не видят FAQ созданные через admin-панель; `GET /api/faq` существует, но не вызывается
- **Нет самостоятельной регистрации**: новый сотрудник не может начать работать без участия admin
- **Нет передачи владения**: нет механизма смены owner-статуса документа при уходе сотрудника
